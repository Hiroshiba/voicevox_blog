# 基本設計

要件を受け取り、コードベースを調査して基本設計を作成する。
コード変更は行わない。

## 処理フロー

1. 要件把握: 与えられた要件や、Issue / PR などの周辺情報を読み、実現したいことを整理する
2. コードベース調査: 要件に関連する既存コード、パターン、制約を調査する。要件定義書があれば読む
3. スコープ定義: 何を変えるかと何を変えないかを明確にする
4. 設計候補の列挙: 実現方法を少なくとも2つ検討し、それぞれのトレードオフを整理する
5. 採用案の決定: 候補から1つを選び、選定理由を明記する
6. 受入基準の定義: 観測可能な振る舞いとして受入基準を記述する
7. 未確定事項の洗い出し: 判断できない事項は黙って埋めず「未定」として残し、判断材料を添える

## 設計の原則

- read-only。コードベースの調査のみ行い変更しない
- 設計候補は必ず複数出す。1案だけで結論しない
- 受入基準は実行可能な検証手段で書く。「仕様通り動く」のような主観的表現を使わない
- 既存パターンとの整合性を確認し、逸脱する場合は理由を明記する
- 未確定事項を仮定で埋めない
- TODO の羅列だけで終えない。レビュアが「進めてよい/だめ」を判断できる粒度にする

## 出力形式

以下の構成で報告する。

### 背景と目的

要件の背景と実現したいことを記述する。

### スコープ

変更範囲と、変更しないものを明記する。

### 現状整理

関連する既存コード、モジュール、パターンを箇条書きで記載する。

### 設計候補

候補ごとに概要とトレードオフを記載する。

### 採用案と理由

選定した候補と、他候補を採用しなかった理由を明記する。

### 受入基準

観測可能な振る舞いとして箇条書きで記載する。

### 未確定事項

判断を保留した事項と判断材料を記載する。該当なしなら「なし」と書く。
