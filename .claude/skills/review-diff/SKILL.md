# 差分レビュープロンプト

ローカルまたは PR の差分をレビューし、指摘と確認済み事項を報告する。

## 処理フロー

1. コンテキスト読み込み: リポジトリに要件定義ファイル・実装計画ファイルがあればそれぞれ必ず読み、プロジェクトの目的と現在の実装方針を把握する。存在しなければこのステップを飛ばす
2. 差分取得:
   - ローカル実行の場合: `git diff` と `git diff --staged` で変更を取得する。`git ls-files --others --exclude-standard` で untracked ファイルも取得し、その内容を読む。すべて空なら「差分なし」と報告して終了する
   - PR 対象の場合: `gh pr diff <番号>` で差分を取得する。空なら「差分なし」と報告して終了する
3. スコープ確認: 変更ファイル一覧を確認し、変更の目的を差分と読み込んだコンテキストから確認する
4. 差分レビュー: 変更されたコードを以下の観点で順に確認する
   - 正当性: ロジックバグ、境界条件、エラーハンドリングの欠落
   - セキュリティ: 秘密値の露出、入力検証の不足、権限昇格
   - 設計: 公開面の変更が意図的か、既存の契約面との不整合
   - 規約準拠: コーディング規約への違反
5. 規約読み込み: リポジトリの規約ファイルを読む。存在しなければこのステップを飛ばす
6. 裏取り: 各指摘について、該当コードと周辺を実際に読み、具体的な失敗シナリオを言語化する。シナリオを言語化できない指摘は CHECKED に分類する
7. 分類: 裏取りを通過した指摘を MUST-FIX / SHOULD-FIX / NOTE に分類する
8. 出力: 分類結果を報告する

## レビューの原則

- 変更されたコードだけをレビューする。変更に含まれないコードの改善提案はしない
- 指摘には必ずファイルパスと行番号を付ける
- 具体的な失敗シナリオを書けない指摘は CHECKED として報告する
- スタイル好み、命名の好み、コメント追加の提案は出さない
- 規約ファイルに明記された規約違反は SHOULD-FIX 以上で出す
- 推測的な指摘は出さない。コードから確認できる事実だけを扱う
- 修正案は書かない

## 分類基準

- MUST-FIX: 正当性バグ、セキュリティ問題、データ損失、公開面の意図しない破壊
- SHOULD-FIX: 規約違反、エラーハンドリング不足、命名と実装のずれ
- NOTE: 将来の変更で問題になりうる設計判断。現時点では壊れない
- CHECKED: 気になって確認したが問題ないと判断したもの。確認した観点と判断理由を記載する

## スコープ制限

- ローカル実行: `git diff`、`git diff --staged`、untracked ファイルに含まれるファイルだけを対象とする
- PR 対象: `gh pr diff` の出力に含まれるファイルだけを対象とする
- diff に含まれない既存コードの問題は指摘しない
- CI 設定、認証設定、インフラ設定の変更は慎重に扱い、判断に迷う場合は NOTE で報告する

## 出力形式

各指摘を以下の形式で報告する:

```
### [MUST-FIX] ファイルパス:行番号

問題の説明（1-2文）

失敗シナリオ: 具体的にどう壊れるか
```

MUST-FIX / SHOULD-FIX / NOTE / CHECKED のいずれも該当しない場合は「レビュー完了。指摘なし」とだけ書く。

## 注意点

数個だけ書くのではなく、見つけたものを全部書く。
