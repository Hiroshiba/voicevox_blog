# タスク分割

詳細設計を受け取り、実装タスクに分割する。
設計の再議論はしない。詳細設計で決まったことを分割するだけ。

## 処理フロー

1. 詳細設計読み込み: 与えられた詳細設計や、Issue / PR などの周辺情報を読み、変更対象・フロー・検証方法を把握する
2. 分割: 検証可能な単位でタスクを分割する。1タスクが1回の実装セッションで完了できる粒度を目安にする
3. 順序決定: 依存関係を考慮して実行順序を決める
4. タスクカード作成: 各タスクの内容を記述する

## 分割の原則

- 1タスク = 1つの検証可能な成果
- 設計を再議論しない。詳細設計の決定に従う
- fresh session がそのタスクカードだけを読んで、どのファイルを触るか・何を成功とするか・どう検証するかを質問せずに実行できる粒度にする
- 依存関係がある場合は本文中で言及する
- 同一ファイルへの変更が競合するタスクは分けない
- 「認証を実装する」のような曖昧なタスクは禁止。具体的な変更単位にする

## タスクカードの要件

各タスクカードの本文には以下を含める:

- ゴール: このタスクで達成すること
- 変更対象: repo-relative フルパス
- 検証方法: 実行コマンドと期待結果
- 完了条件: 観測可能な振る舞いとして記述する。「仕様通り動く」は禁止

依存関係がある場合はその旨を本文に記載する。
