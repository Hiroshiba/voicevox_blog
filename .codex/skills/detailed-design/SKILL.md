# 詳細設計

基本設計を受け取り、実装に必要な詳細を具体化する。
コード変更は行わない。

## 処理フロー

1. 基本設計読み込み: 与えられた基本設計や、Issue / PR などの周辺情報を読み、採用案と受入基準を把握する
2. コードベース調査: 変更対象のファイルと周辺コードを読んで確認し、現在の実装を把握する
3. 変更対象の特定: 変更が必要なファイルを repo-relative フルパスで列挙する。新規ファイルも含める
4. 変更内容の具体化: 各ファイルの変更責務を記述する。追加・変更する関数・クラス・データ構造を列挙する
5. フロー記述: 正常系と異常系の処理フローを記述する
6. 検証方法の定義: 検証コマンドと期待結果を明記する。どのディレクトリで何を実行し何が出れば成功かを書く
7. 影響範囲の確認: 変更が他の機能に与える影響を確認する

## 設計の原則

- read-only。コードベースの調査のみ行い変更しない
- fresh session がこの文書だけで実装着手できるレベルまで落とす。外部文書参照に逃げず必要情報をこの文書に埋め込む
- 変更対象は repo-relative フルパスで列挙する。曖昧な参照をしない
- 用語は repo 内の実体に結び付けて説明する
- 責務・制約・検証を固定し、実装の細かい書き方は縛らない
- 検証コマンドは実行可能な形式で書く。「テストを通す」のような曖昧な表現を使わない

## 出力形式

以下の構成で報告する。

### 前提

基本設計の採用案を1-2文で要約する。

### 変更対象ファイル

ファイルごとに以下を記載する:
- repo-relative フルパス
- 変更責務
- 追加・変更する関数・クラス・データ構造

新規ファイルも同様に記載する。

### 正常系フロー

処理の流れを番号付きで記述する。

### 異常系フロー

エラーケースごとの処理を記述する。

### 制約

実装が守るべき制約を箇条書きで記載する。

### 検証方法

検証コマンドと期待結果を記載する。

### 影響範囲

変更が他の機能に与える影響を記載する。該当なしなら「なし」と書く。
